## Introduction

This feature allows you to define your database schema by using the code-first approach and automatically generate your migrations
by identifying drifts between your models and your database.

```ts tab="Model"
export default class Post extends BaseModel {
  @column.uuid({ isPrimary: true })
  declare id: string

  @column.string()
  declare title: string

  @column.text({ nullable: true })
  declare description: string | null

  @column.updatedAt()
  declare updatedAt: DateTime

  @column.createdAt()
  declare createdAt: DateTime
}
```

```ts tab="Migration"
export default class extends BaseSchema {
  async up() {
    this.createTable('users', (table) => {
      table.uuid('id').primary().notNullable()
      table.string('title').notNullable()
      table.text('description').nullable()
      table.datetime('updated_at').notNullable()
      table.datetime('created_at').notNullable()
    })
  }

  async down() {
    this.dropTable('users')
  }
}
```

### Philosophy

The primary goal of ORMs (Object-Relational Mapper) is to map database tables and rows to objects in your code.

To achieve this you need to define both your models (code) and your database schema (schema) which is redundant as the relationship is almost 1 to 1.
This is the reason why most ORMs provide a way to automatically **generate your models using your database schema (schema-first)** or your **database schema base on your models (code-first)**.

Code-first vs schema-first is a matter of preferences and usage but most of industry choosed to use the **code-first** approach.

| ORM                                                               | Type         |
| ----------------------------------------------------------------- | ------------ |
| [Drizzle](https://orm.drizzle.team/)                              | Code-first   |
| [Prisma](https://www.prisma.io/)                                  | Code-first   |
| [Doctrine](https://www.doctrine-project.org/)                     | Code-first   |
| [TypeORM](https://typeorm.io/)                                    | Code-first   |
| [Sequelize](https://sequelize.org/)                               | Code-first   |
| [MikroORM](https://github.com/mikro-orm/mikro-orm)                | Code-first   |
| [Lucid](https://lucid.adonisjs.com/)                              | Schema-first |
| [Kysely Codegen](https://github.com/RobinBlomberg/kysely-codegen) | Schema-first |

<Callout>
  Other well-known ORMs like
  [ActiveRecords](https://guides.rubyonrails.org/active_record_basics.html),
  [Hybernate](https://hibernate.org/) or [Eloquent](https://laravel.com/docs/12.x/eloquent) does not
  have such feature and usually simply rely on "make" commands and will not help you with schema
  changes.
</Callout>

#### Why code-first

- **Models are simpler to write**  
  Models feel idiomatic and easy to reason about while migrations are usually hard to write and read.
- **Faster iteration**  
  Change code -> run migration -> done
- **Detect drifts**  
  Block deployments when there are differences between your new database schema and models.
- **Customizable**  
  Generated migrations can be modified
- **Migrations squashing**  
  To avoid the migrations hell you can simply delete all your migrations and recreate them all in a single file
- **Improved models metadata**  
  Libraries relying on metadata ([@foadonis/openapi](../openapi) or [@foadonis/graphql](../graphql)) can use schema informations.
