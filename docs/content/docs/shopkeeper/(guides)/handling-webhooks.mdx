Stripe uses [webhooks](https://stripe.com/docs/webhooks) to notify your application when events happen, like a subscription being cancelled or a payment failing. Shopkeeper listens to these events and automatically keeps your database in sync.

## How it works

Shopkeeper registers a webhook endpoint at `/stripe/webhook` that receives events from Stripe. Out of the box, it handles:

- Subscription creation, updates, and deletion
- Customer updates and deletion
- Payment method changes
- Failed payment retries and invoice events

You don't need to write any code for these. Shopkeeper processes the events and updates your database accordingly.

## Required events

For Shopkeeper to work correctly, the following webhook events must be enabled in Stripe:

- `customer.subscription.created`
- `customer.subscription.updated`
- `customer.subscription.deleted`
- `customer.updated`
- `customer.deleted`
- `payment_method.automatically_updated`
- `invoice.payment_action_required`
- `invoice.payment_succeeded`

## Creating the webhook

The easiest way to configure the webhook in Stripe is with the `shopkeeper:webhook` Ace command. It creates a webhook with all the required events:

```sh
node ace shopkeeper:webhook
```

By default, the webhook URL is built from your `APP_URL` environment variable. You can override it:

```sh
node ace shopkeeper:webhook --url "https://example.com/stripe/webhook"
```

Other options:

```sh
# Use a specific Stripe API version
node ace shopkeeper:webhook --api-version="2024-12-18.acacia"

# Create the webhook in a disabled state
node ace shopkeeper:webhook --disabled
```

## Testing locally

During development, use the [Stripe CLI](https://docs.stripe.com/stripe-cli) to forward webhook events to your local application:

```sh
stripe listen --forward-to localhost:3333/stripe/webhook
```

The CLI will output a webhook signing secret starting with `whsec_`. Add it to your `.env` file:

```ini title=".env"
STRIPE_WEBHOOK_SECRET=whsec_xxxxxxx
```

You can then trigger test events with:

```sh
stripe trigger customer.subscription.created
```

## Handling custom events

If you need to react to events beyond what Shopkeeper handles automatically, you can listen to them via the AdonisJS [Event Emitter](https://docs.adonisjs.com/guides/digging-deeper/emitter).

Shopkeeper emits events using the naming convention `stripe:<eventType>`:

```ts title="start/events.ts"
import emitter from '@adonisjs/core/services/emitter'
import type Stripe from 'stripe'

emitter.on(
  'stripe:checkout.session.completed',
  (event: Stripe.CheckoutSessionCompletedEvent) => {
    // Fulfill the order, send a confirmation email, etc.
  }
)
```

Events with the `:handled` suffix are emitted after Shopkeeper has finished processing the event. This is useful when you need to run logic that depends on the database being up to date:

```ts title="start/events.ts"
emitter.on(
  'stripe:customer.subscription.created:handled',
  (event: Stripe.CustomerSubscriptionCreatedEvent) => {
    // The subscription is now saved in your database
  }
)
```

## Idempotent handling

Shopkeeper stores processed webhook events in the `stripe_webhook_events` table to prevent duplicate processing. If Stripe retries a webhook delivery, Shopkeeper will recognize the event and skip it.

This is handled automatically. The `stripe_webhook_events` table is created by the migration included with Shopkeeper.

## Webhook signature verification

Shopkeeper automatically verifies the signature of incoming webhook requests to ensure they are authentic. This requires the `STRIPE_WEBHOOK_SECRET` environment variable to be set.

<Callout type="warn">
The `STRIPE_WEBHOOK_SECRET` is mandatory in production. Without it, your application cannot verify that incoming webhook requests are genuinely from Stripe.
</Callout>
